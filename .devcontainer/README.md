# Dev container config

This folder contains the configuration to work in a dev container: an isolated, reproducible development environment that runs inside Docker.

The configuration skeleton is created by the Guardian's [devenv](https://github.com/guardian/devenv) config tool.

The devenv tool will be available to you using `mise` locally (if you run `mise install`), and automatically in the container.

## Benefits of working in a dev container

* **Consistent environment**: Every developer gets the same OS, JVM, and toolchain versions, eliminating "works on my machine" problems.
* **Zero local setup**: No need to install Java, sbt, or other tools manually. The container handles everything on first start.
* **Shared dependency caches**: Coursier and Ivy caches are stored in named Docker volumes (`devenv-coursier-cache-volume`, `devenv-ivy-cache-volume`) that persist across container rebuilds and are shared between projects using the same devenv setup, so dependency downloads are fast.
* **IDE integration**: Pre-configured extensions and plugins for VS Code and IntelliJ are installed automatically.
* **Isolation**: Project tooling doesn't pollute your host machine.

## Contents

### `devenv.yaml`

The source of truth configuration for this project's dev container. Edit this file to add/remove devenv modules, then run `devenv generate` to regenerate the shared `devcontainer.json` file.

### `shared/devcontainer.json`

The main devcontainer configuration, committed to the repository and used by all developers. It is generated from `devenv.yaml` and defines:

* Container image
* VS Code extensions installed automatically
* JetBrains plugins installed automatically
* Lifecycle hooks:
  * `onCreateCommand`: symlinks the shared Coursier and Ivy cache volumes into the expected locations inside the container.
  * `postCreateCommand`: installs and configures mise, then installs all project tooling declared in the mise config.

Do not edit this file directly; edit `devenv.yaml` and regenerate.

### `user/devcontainer.json`

A personal copy of the devcontainer config, **git-ignored** (see `.gitignore`). This file is intended for your own overrides: extra VS Code extensions, personal settings, or tweaks that shouldn't affect other contributors. Modify this file freely without worrying about polluting shared config.

## Using the dev container

### VS Code

1. Install the [Dev Containers extension](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers).
2. Open the repository in VS Code, then when prompted click **Reopen in Container**, or run the command `Dev Containers: Reopen in Container` from the command palette.
3. VS Code will build the container, run the setup scripts, and install extensions automatically.

Full documentation: https://code.visualstudio.com/docs/devcontainers/containers

### IntelliJ IDEA

IntelliJ supports dev containers via [JetBrains Gateway](https://www.jetbrains.com/remote-development/gateway/).

1. Use **File → Remote Development → Dev Containers**.
2. Gateway will start the container and open IntelliJ connected to it, with the Scala plugin pre-installed.

Full documentation: https://www.jetbrains.com/help/idea/connect-to-devcontainer.html

## Current limitations

* **Cannot push from Intellij**: There's an open bug working with ssh on Intellij's system.
* **First-start time**: The initial container build downloads the base image, installs mise, and downloads JVM dependencies. This can take several minutes. Subsequent starts are much faster due to cached volumes.

## To be aware of
* **Docker required**: You must have Docker Desktop running on your laptop.
* **Resource usage**: The container shares your host's CPU and RAM. Large Scala builds can be memory-intensive. Ensure Docker is configured with sufficient memory (4 GB+ recommended).  This is set in user/devcontainer.json.
